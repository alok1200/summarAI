   176→Stage Summary:
   177→- Long videos (up to 50 hours) now work end-to-end. The pipeline is:
   178→  1. Chunk: ~22K-char chunks at segment boundaries (preserves timestamps)
   179→  2. Map: parallel per-chunk LLM calls (4-way concurrency) with per-chunk error isolation
   180→  3. Reduce: single LLM call that merges per-chunk results into a coherent final answer
   181→  4. Stream: progress indicators + final answer stream to the client in real time
   182→- Speed: parallel processing gives ~4x speedup over sequential. A 73K-char transcript summary completes in 15s (vs 60+ seconds sequential). First byte always <1s, so no proxy 502s.
   183→- Quality: for interview Q&A, the reduce step is explicitly told to pick DIVERSE questions spanning all chunks (not cluster around one segment). Test confirms questions come from timestamps across all 4 chunks.
   184→- Q&A mode: retrieval-augmented. For each user question, an LLM picks the top 3 most relevant chunks (from the topic index), and only those chunks are injected into the context. Off-topic questions short-circuit instantly with the canned reply, no main LLM call needed.

   760→=== (3) SQLite BACKUP STORY — VACUUM INTO + cron-friendly CLI ===
   761→
   762→- New script: scripts/db-backup.ts (uses `bun:sqlite`, no extra deps).
   763→  Commands:
   764→    bun run scripts/db-backup.ts                   # one-shot backup
   765→    bun run scripts/db-backup.ts --list            # list backups (newest first)
   766→    bun run scripts/db-backup.ts --prune           # delete beyond retention
   767→    bun run scripts/db-backup.ts --restore <name>  # restore (with safety backup)
   768→  Env vars (all optional, all documented in .env.example):
   769→    DATABASE_URL       (default: file:./db/custom.db)
   770→    BACKUP_DIR         (default: ./backups)
   771→    BACKUP_RETENTION   (default: 30)
   772→    BACKUP_COMPRESS    (default: "1" = gzip)
   773→- Added npm scripts: db:backup, db:backup:list, db:backup:prune, db:restore.
   774→- WHY VACUUM INTO, not `cp`:
   775→  SQLite uses WAL mode by default; a raw `cp db/custom.db backups/` can
   776→  capture the DB in an inconsistent state if a write is in flight.
   777→  `VACUUM INTO 'path'` is SQLite's built-in Online Backup mechanism
   778→  (since 3.27, 2019) — produces a transactionally-consistent snapshot
   779→  even while the DB is being written to. Same guarantee as `sqlite3
   780→  .backup`, no shell-out needed. Source is opened read-only so we don't
   781→  conflict with the running Prisma server.
   782→- Restore creates a "pre-restore" safety backup BEFORE overwriting the
   783→  live DB, so a botched restore is always recoverable.
   784→- Auto-prune: after each backup, deletes oldest backups beyond
   785→  BACKUP_RETENTION count. Cron-friendly — recommended setup documented
   786→  in .env.example:
   787→    0 3 * * *  cd /path/to/app && npm run db:backup >> /var/log/db-backup.log 2>&1
   788→- Added /backups/ to .gitignore.
   789→- Smoke-tested all 4 commands end-to-end:
   790→  * backup → 5.5 KB gzipped, 3-5ms
   791→  * list → shows timestamped entries newest-first
   792→  * restore → creates pre-restore safety backup, then overwrites live DB
   793→  * prune (BACKUP_RETENTION=2) → deletes oldest, keeps newest 2
